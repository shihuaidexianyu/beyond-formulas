\section{P、NP 与 NP-hard：给问题按计算难度分类}
\label{sec:09-Convex-Optimization/09-Complexity-and-Approximation/01}

\begin{quote}
\textbf{先抓住这一点：}NP-hard 说的是一整类问题随输入规模增长时的最坏情形，不是说每个实例都无解或实际中一定很慢。复杂度分类帮助我们判断该继续找精确算法，还是转向近似、特殊结构与启发式方法。
\end{quote}

判断一个问题``难不难''，不能只凭直觉说``看起来复杂''。计算复杂度理论把问题按最坏情况下所需资源分类，让我们能够精确地区分``理论上可解''与``实际上可算''。

\subsection{P：多项式时间可解}

一个判定问题（答案只有``是''或``否''）属于 P，如果存在确定性算法，在输入长度 \(n\) 的多项式时间内给出答案。排序、最短路径、线性规划（弱多项式意义下）、二分图匹配都属于 P。

P 的关键性质是组合封闭性：P 中问题的多项式次调用仍在 P 中。如果问题 A 多项式时间归约到问题 B（\(A\le_p B\)），且 \(B\in\) P，则 \(A\in\) P。这让我们能够把新问题``翻译''成已知可解的问题。

多项式时间是``可解''的最低标准，不是高效标准。\(O(n^{100})\) 的算法在 P 中，但毫无实用价值。P 的意义在于划定理论边界：不在 P 中的问题，大规模实例上通常不可行。

\subsection{NP：多项式时间可验证}

NP 包含所有``答案为是时，存在多项式长度的证书，可以在多项式时间内验证''的问题。图的哈密顿回路：给一个顶点排列，多项式时间就能检查它是否构成哈密顿回路。整数规划的可行性：给一组整数赋值，检查约束是否满足只需代入计算。

P \(\subseteq\) NP：可解必然可验证（证书就是空，直接算）。反过来，NP \(\stackrel{?}{=}\) P 是计算理论最著名的未解问题。普遍猜测是 P \(\ne\) NP，即存在可验证但不可快速求解的问题。

NP 的定义只要求``是''的实例可验证，不要求``否''的实例可验证。coNP 是 NP 的补类，处理``否''的证书。NP \(\cap\) coNP 中的问题（如线性规划、素数判定）两个方向都有短证书，通常被认为比 NP-hard 问题``容易''。

\subsection{NP-hard 与 NP-complete}

问题 \(B\) 是 NP-hard，如果所有 NP 问题都能多项式时间归约到它：\(A\le_p B\) 对所有 \(A\in\) NP 成立。这意味着 \(B\) 至少和 NP 中最难的问题一样难——若 \(B\) 有多项式算法，则 P = NP。

NP-complete 是 NP-hard 且属于 NP 的问题。它们是 NP 中最难的问题，包括：

\begin{itemize}
\tightlist
\item
  可满足性（SAT）：第一个被证明的 NP-complete 问题（Cook--Levin 定理）；
\item
  旅行商问题（TSP）：访问所有城市恰好一次的最短回路；
\item
  整数规划：线性约束下的整数优化；
\item
  最大割：把图的顶点分成两组，使跨组边数最大；
\item
  图着色：用最少的颜色给顶点染色，使相邻顶点不同色。
\end{itemize}

证明一个问题是 NP-hard 的标准方法是归约：从已知 NP-hard 问题 \(A\) 构造多项式时间变换，把 \(A\) 的实例映射到 \(B\) 的实例，保持``是''与``否''的对应。一旦建立了 NP-hard 地位，问题就被认为``本质困难''——除非 P = NP，否则没有多项式精确算法。

\subsection{NP-hard 问题的三种应对策略}

面对 NP-hard 问题，实践中有三条路：

\begin{enumerate}
\tightlist
\item
  \textbf{精确算法}：接受指数时间，但优化常数（分支限界、动态规划、问题分解）。适合小规模实例或结构良好的特例。
\item
  \textbf{启发式}：放弃最坏情况保证，用局部搜索、模拟退火、遗传算法等寻找``通常不错''的解。风险是无法预知何时会严重失败。
\item
  \textbf{近似算法}：多项式时间内给出解，并证明其目标值与最优值的比值有最坏情况保证。这是本单元的主题。
\end{enumerate}

近似算法的价值在于可证明的边界：不是``通常好用''，而是``最坏情况下也有保证''。这种保证让我们能够像对待凸优化那样对待 NP-hard 问题——知道解离最优有多远。

\subsection{复杂度不是唯一的困难维度}

NP-hard 是最坏情况分类，不意味着所有实例都难。随机实例、参数化实例（固定某个参数后多项式）、以及``典型''实例可能容易得多。参数化复杂度（parameterized complexity）和平滑分析（smoothed analysis）研究这些更细的结构。

复杂度也不等于统计难度。一个问题可以计算上困难但统计上容易（需要的数据少），也可以计算上容易但统计上困难（需要的数据多）。Part 13 的学习理论处理统计复杂度，本单元处理计算复杂度——两者独立，不能互相替代。
