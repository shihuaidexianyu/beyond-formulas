\section{Huffman 编码与贪心最优性}

Kraft 不等式告诉我们哪些长度组合可行，却没有告诉我们怎样针对给定概率选择最佳长度。若概率已知，我们希望最小化

\[
L=\sum_xp(x)\ell(x).
\]

直觉上，高概率符号应更靠近树根，低概率符号应更深。但树的分支必须整体安排，不能独立地把每个 \(-\log_2p(x)\) 四舍五入。Huffman 算法的价值，是把这个全局树优化问题化成一连串局部合并。

\subsection{从树的最深处反推}

最优码树必为满树：若某内部节点只有一个孩子，把该孩子的子树上移一层，部分码字变短而无一变长，与最优矛盾。满树中最深叶的同胞必为同深度的叶——它若是内部节点，就会产生更深的叶——因此最深的两个叶节点可以取为同胞。若它们不是概率最小的两个符号，我们可以把较小概率的符号与更深位置的较大概率符号交换。码长变化对平均代价的影响为

\[
(p_{\text{small}}-p_{\text{large}})
(\ell_{\text{deep}}-\ell_{\text{shallow}})\le0,
\]

所以交换不会使代价增加。因此总存在一棵最优树，使概率最小的两个符号位于最深层并共享父节点。

把这两个符号暂时合并成一个``复合符号''，其概率等于二者之和，原问题就缩成少一个符号的同类问题。复合符号在缩约树中的码长若为 \(r\)，拆开后两个子符号长度都是 \(r+1\)；总代价正好多出它们的概率和。

这给出递归算法：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  取当前概率最小的两个节点；
\item
  合并成权重为二者之和的新节点；
\item
  把新节点放回集合，重复直到只剩根；
\item
  从根沿左右边标记 0 和 1，读出每个叶的码字。
\end{enumerate}

\subsection{完整算一个例子}

设

\[
p(A)=0.4,\quad p(B)=0.3,\quad p(C)=0.2,\quad p(D)=0.1.
\]

首先合并 \(D\) 与 \(C\)，得到权重 \(0.3\)；现在有 \(0.3,0.3,0.4\)。再合并两个 \(0.3\)，得到 \(0.6\)；最后与 \(0.4\) 合并。

一种可能的码是

\[
A\mapsto0,\quad
B\mapsto10,\quad
C\mapsto110,\quad
D\mapsto111.
\]

平均码长为

\[
L=0.4\cdot1+0.3\cdot2+0.2\cdot3+0.1\cdot3=1.9\text{ bit}.
\]

而熵约为

\[
H(X)\approx1.846\text{ bit}.
\]

差距只有约 \(0.054\) bit。并列权重可能产生不同树和不同码字，但它们的平均长度可以相同；码字本身并不唯一，最优值才是算法保证的对象。

\subsection{贪心为什么真的全局最优}

很多贪心算法看起来合理却会失败，Huffman 需要一个明确证明。证明由两部分组成：

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  交换论证：存在一棵最优树，其中最小的两个概率处于最深同胞位置；
\item
  最优子结构：把这两个叶合并后，剩余树必须是缩约问题的最优树，否则用更优缩约树替换，就能让原树更优，产生矛盾。
\end{enumerate}

对符号数归纳，反复合并最小权重便得到全局最优的二元逐符号前缀码。这一结论的限定词都很重要：有限字母表、已知概率、二元、逐符号、前缀码、最小期望长度。

\subsection{熵界与等号}

Huffman 码同样满足

\[
H(X)\le L_{\mathrm H}<H(X)+1.
\]

左边是所有唯一可译码共有的熵下界；右边可由上一节 Shannon 码的存在性以及 Huffman 在前缀码中的最优性得到。

若概率为 dyadic 形式，例如

\[
(1/2,1/4,1/8,1/8),
\]

理想长度 \((1,2,3,3)\) 都是整数，Huffman 可达到 \(L=H(X)=1.75\) bit。反之，非 dyadic 概率通常无法逐符号精确达到熵，因为树深必须是整数。

\subsection{Huffman 最优，却仍可能显得很浪费}

考虑 Bernoulli\((0.99)\) 信源。单符号二元前缀码只有 \(0\) 和 \(1\)，无论概率多偏，平均仍是 1 bit；但熵只有约 \(0.081\) bit。Huffman 已经在自己的问题类别中最优，差距却很大。

问题出在编码类别限制得过窄，算法本身仍然正确。若把 100 个符号组成一块，常见序列可以共享整数码长的舍入误差；若直接为整段序列分配区间，还能更平滑地累积分数 bit。下一节的算术编码正是后一种思想。

\subsection{实现和模型的边界}

真实压缩器还需要处理算法定理之外的成本：

\begin{itemize}
\tightlist
\item
  概率表或码树本身如何传给解码器；
\item
  经验分布是否代表未来数据；
\item
  概率并列时如何保证编码端与解码端采用相同规则；
\item
  \(D\) 元 Huffman 中是否需要补零权重，使叶数满足树结构要求；
\item
  单符号字母表是否允许空码字。
\end{itemize}

理论平均码长不包含这些协议开销。小文件中，码表可能比被压缩内容还大。
